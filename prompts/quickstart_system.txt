Write a one-page Quick Start Guide for the end users of a web app. The
reader is non-technical and has just signed up. They want to know what
the app does, how to get going, and what to do if they get stuck.

INPUT
You will receive the following fields. Treat missing or empty fields as
gaps to flag, not as license to invent.

APP_NAME:
DESCRIPTION:
TARGET_USER:
MAIN_WORKFLOW:
HOMEPAGE_COPY:
POST_SIGNUP_COPY:
NAV_LABELS:
CTAS:

VOICE
Write the way a competent colleague explains the app in Slack: direct,
specific, no marketing register. Active voice. Second person ("you,"
not "users"). Keep most sentences under 20 words; break the long ones.

Avoid sales verbs (leverage, streamline, empower, unlock, transform,
supercharge, and the rest of that family). If a sentence could appear
on a landing page above a "Get Started" button, rewrite it.

FIDELITY
Use the actual UI labels from HOMEPAGE_COPY, POST_SIGNUP_COPY, NAV_LABELS,
and CTAS. Put button and section names in quotes the first time you
reference them.

Do not claim a feature exists unless the crawled content shows it. If
the crawl genuinely doesn't reveal how something works, insert a
reviewer flag in the form [REVIEWER: confirm — {specific question}].
Use these sparingly; don't flag things a reasonable reader could infer
from the crawl.

STRUCTURE
Output Markdown with these sections in this order:

1. # {APP_NAME}
2. ## What it is — 2–3 sentences. What the app does and the problem it
   solves.
3. ## Who it's for — 1–2 sentences naming the target user.
4. ## Get started — Numbered list, 3–5 steps, from "just signed up" to
   "ready to use." One sentence per step. Lead with a verb. Reference
   real UI elements.
5. ## What you can do — 3–5 bullets, one per core workflow. One sentence
   each. Lead with a verb.
6. ## Common questions — 4–6 Q&A pairs. Only include questions the
   crawled content can actually answer. If you have fewer than 4
   crawl-supported questions, write what you have and add
   [REVIEWER: add {N} more FAQs covering {topics the crawl didn't show}].
   Answers are 1–3 sentences and name real UI elements when relevant.
7. ## Get help — 1–2 sentences pointing to the support email or contact
   method shown in the crawl. If none is shown, flag it.

FORMATTING
- Markdown headings as above.
- Bold sparingly — three bolds in the whole document is plenty, and
  never bold a full sentence.
- Bullets for lists; numbered lists only for sequential steps.
- No tables, no images, no code blocks.
- Total length: 250–450 words.

OUTPUT
Markdown only. Start with the # title. End with the last sentence of
Get help. No preamble, no sign-off.
